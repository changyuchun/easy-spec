# easy-spec


`easy-spec` 是一个**开发全周期规格管理 skill**。  
从一句话需求出发，通过连续追问生成规格文档和任务清单，并维护从"规格确定"到"需求交付归档"的完整开发上下文。

## 目录结构

所有文件统一在 `.easy-spec/` 下，不散落到项目根目录。

```
项目根目录/
└── .easy-spec/
    ├── spec/                              ← 当前活跃需求（工作台）
    │   ├── Spec.md                        ← 规格文档（核心产出）
    │   ├── Task-List.md                   ← 开发任务清单
    │   └── Spec-CHANGELOG.md             ← 变更记录（按需）
    ├── memory/
    │   ├── context.md                     ← Level 0：当前会话进度
    │   ├── index.md                       ← Level 1：所有历史需求索引
    │   └── summaries/                     ← Level 2：每个需求的概要
    │       ├── 用户标签系统.md
    │       └── 订单状态机重构.md
    └── archive/                           ← Level 3：完整归档（只读）
        └── 2026-07-03-用户标签系统/
            ├── Spec.md
            ├── Task-List.md
            └── retro.md
```

### 四级记忆体系

| 级别 | 文件 | 内容 | 加载时机 |
|------|------|------|---------|
| Level 0 | `memory/context.md` | 当前任务进度、阻塞、下一步 | 每次会话启动**自动** |
| Level 1 | `memory/index.md` | 所有历史需求一行索引 | 每次会话启动**自动** |
| Level 2 | `memory/summaries/<名>.md` | 需求概要：决策 + 实体 + 技术债 | 发现相关需求时**按需** |
| Level 3 | `archive/<日期-名>/` | 完整 Spec + Task-List + retro | 用户显式要求时**显式** |

## 触发方式

```
# 新需求：
使用 easy-spec，帮我把这个需求做成规格：[一句话需求]

# 继续已有规格：
使用 easy-spec，在现有 Spec.md 基础上迭代

# 每次会话启动：
请读取 .easy-spec/memory/context.md，告诉我当前进度，今天从哪里开始

# 需求完成归档：
这个需求做完了，请使用 easy-spec 帮我归档
```

## 工作流总览

```
[启动]  自动加载 Level 0 context.md + Level 1 index.md
        ↓ 扫描 index.md，发现相关历史需求 → 询问是否加载 Level 2 摘要
        ↓
Step 0  展示探讨计划，等用户确认
        ↓
Step 1  确认需求名称与核心骨架（每轮 3 个问题）
        ↓
Step 2  建立问题矩阵，逐域追问
        ↓
Step 3  输出 Spec.md
        ↓
Step 4  输出 Task-List.md
        ↓
Step 5  初始化 context.md
        ↓
        ⬇ 日常开发：每次会话读 context.md + index.md，更新进度
        ↓
Step 6  归档：Level 3 archive/ + 更新 Level 1 index.md + 生成 Level 2 summary
               清空 context.md
```

## 完整示例：用户标签系统

### 初始输入

```
使用 easy-spec，帮我把这个需求做成规格：
运营可以给用户手动打标签，推送系统按标签筛选用户。
```

---

### Step 0：探讨计划

**AI 输出**（等用户确认再开始）：

```
本次探讨计划：

1. 确认需求名称
2. 厘清打标签的规则（谁打、打几个、标签从哪里来）
3. 确认筛选的逻辑（AND / OR、实时 / 延迟）
4. 权限设计（谁能创建标签、谁能打标签）
5. 异常边界（删标签后怎么处理）
6. 确认是否需要生成 Task List

预计需要 3-4 轮追问，每轮 3 个问题。

确认后开始，或告诉我需要调整哪些方向。
```

用户：确认，开始。

---

### Step 1：第 1 轮追问（核心骨架）

**AI 问**：
```
1. 这个需求的工作名是什么？（如"用户标签系统"）
2. 标签是运营从固定词库里选，还是可以自定义创建？
3. 这个需求需要生成开发任务清单吗？
```

**用户答**：
```
1. 就叫用户标签系统
2. 运营可以自定义创建，但要过审（超级运营审核）
3. 需要，我们要直接开发
```

**AI 内部**：更新 Spec.md 草稿，标记"标签审核流程"已确认，"标签创建权限"已确认。  
**AI 对用户同步**（轻量）：已锁定：需求名称、标签需自定义创建 + 审核。待问：筛选逻辑、权限细节、边界处理。

---

### Step 1：第 2 轮追问（规则）

**AI 问**：
```
1. 按标签筛选用户时，支持多标签组合（AND / OR）吗？
2. 一个用户最多可以打多少个标签？
3. 用户数量大约是什么量级？（影响筛选方案）
```

**用户答**：
```
1. 支持 AND 和 OR 组合
2. 最多 20 个
3. 大约 500 万用户
```

---

### Step 1：第 3 轮追问（权限与边界）

**AI 问**：
```
打标签权限是以下哪种方式？
A. 普通运营只能用已审核标签打标；超级运营可以创建 + 审核
B. 普通运营可以创建草稿，超级运营审核后生效

2. 标签被删除（禁用）后，已打到用户身上的标签怎么处理？
3. 推送系统是直接查 DB 还是调你们的接口？
```

**用户答**：
```
1. A
2. 软删除，历史保留，但新筛选不匹配已禁用标签
3. 调我们的 REST 接口
```

**AI 内部**：所有问题矩阵已覆盖，准备输出 Spec.md。

---

### Step 3 产出：`.easy-spec/spec/Spec.md`

```markdown
# Spec — 用户标签系统

## 1. 原始输入
- 需求名称：用户标签系统
- 一句话需求：运营可以给用户手动打标签，推送系统按标签筛选用户

## 2. 需求概述
- 一句话定位：运营手动标签管理 + 用户标签筛选接口
- 主要使用方：普通运营（打标签）、超级运营（审核标签）、推送系统（调用筛选接口）
- 一期范围：标签 CRUD（需审核）、打标签、按标签筛选用户接口

## 4. 使用方与场景
### 主要使用方
- 普通运营：给用户打标签（只能使用已审核标签）
- 超级运营：创建标签 + 审核标签
- 推送系统：调用筛选接口获取目标用户列表

### 核心场景
- 超级运营创建"高价值用户"标签并审核通过
- 普通运营在用户详情页给用户打上"高价值用户"标签
- 推送系统调用接口，查询同时有标签 A 且有标签 B 的用户

## 6. 核心对象与状态

### Tag（标签）
- 状态：待审核 → 已启用 → 已禁用
- 流转：创建→待审核（自动）；超级运营审核→已启用；超级运营禁用→已禁用
- 禁用为软删除，记录保留

### UserTag（用户-标签关系）
- 一个用户最多 20 个标签（应用层校验）
- 标签禁用后，历史 UserTag 保留但不参与筛选

## 8. 业务规则
- 普通运营只能使用 status=ACTIVE 的标签打标，否则报错
- 标签创建只有超级运营可操作
- 筛选支持多标签 AND / OR 逻辑组合
- 500 万用户量级的筛选接口 P99 ≤ 2 秒

## 14. 验收标准
- 普通运营可以给用户打标签，打完立即可查询
- 超级运营审核通过后标签对普通运营可见
- AND 条件筛选 500 万用户 P99 ≤ 2 秒
- 禁用标签后，筛选接口不返回该标签下的用户
```

## 每个步骤汇总

| 步骤 | 触发 | 输入 | 产出文件 |
|------|------|------|---------|
| 启动加载 | `使用 easy-spec` 或 `读取 context.md` | — | 读取 `context.md`（L0）+ `index.md`（L1）|
| 历史关联感知 | index.md 中发现相关需求 | index.md | 询问用户，确认后读取 `summaries/<名>.md`（L2）|
| 探讨计划 | 用户确认启动 | 一句话需求 + L2 背景（如有）| 无（等用户确认）|
| 追问 | 用户确认计划 | 每轮用户回答 | 无（内部草稿）|
| 输出 Spec | 问题矩阵覆盖完成 | 追问结果 | `spec/Spec.md` |
| 输出 Task List | Spec 完成 + 需要执行 | Spec.md | `spec/Task-List.md` |
| 初始化上下文 | 进入开发阶段 | Spec + Task-List | `memory/context.md` |
| 日常会话更新 | 用户指令 | 当前进度 | `context.md`（更新）|
| 归档 Step 1 | 用户说"做完了" | 规格文件 | `archive/<日期-名>/`（L3）|
| 归档 Step 2 | 归档触发 | retro 复盘信息 | `archive/<日期-名>/retro.md` |
| 归档 Step 3 | 归档触发 | 需求信息 | `memory/index.md`（追加一行，L1）|
| 归档 Step 4 | 归档触发 | Spec + retro | `memory/summaries/<名>.md`（L2）|
| 归档 Step 5 | 归档触发 | — | `context.md`（清空）|
