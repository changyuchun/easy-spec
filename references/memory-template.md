# 记忆系统设计模板

描述 easy-spec 的两层记忆体系，以及如何在 Claude Code 全局记忆中沉淀跨需求经验。

---

## 两层记忆体系

### 第一层：项目级记忆（`.easy-spec/memory/context.md`）

**定位**：当前活跃需求的开发上下文快照

**生命周期**：
- 由 easy-spec 在 Task List 生成后创建
- 每次会话结束时由用户或 AI 更新
- 需求归档时清空，重置为空模板

**内容范围**：
- 当前需求名称与阶段
- 任务进度（已完成 / 进行中 / 待开始）
- 已知阻塞点
- 最近 5 条关键决策
- 下一步建议

**不写入**：
- Spec.md 和 Task-List.md 中已有的完整内容
- 代码实现细节
- 个人学习笔记或临时想法

---

### 第二层：Claude Code 全局记忆（`~/.claude/projects/.../memory/`）

**定位**：跨需求、跨会话的持久经验，由 Claude 自动写入

**类型**：

| 类型 | 写入时机 | 示例 |
|------|---------|------|
| `user` | 了解用户角色和偏好 | 用户是 Java 架构师，不需要解释 Spring 基础 |
| `feedback` | 用户纠正行为或确认非显而易见的做法 | 测试不能 mock 数据库，必须打真实 DB |
| `project` | 了解项目关键上下文 | merge freeze 在 7 月 1 日 |
| `reference` | 发现重要外部资源位置 | bug 追踪在 Linear project "BACKEND" |

**写入原则**：
- 只写跨需求有持久价值的内容
- 不写当前需求的进度状态（那是 context.md 的职责）
- 不写代码结构和文件路径（可以从代码库直接推断）
- 不写 git 历史相关内容（git log 更权威）

---

## 记忆读取顺序

每次会话启动，按以下顺序读取（由用户在第一条消息中触发）：

```
1. .easy-spec/memory/context.md       ← 当前需求状态
2. .easy-spec/spec/Spec.md            ← 需要理解规格细节时
3. .easy-spec/spec/Task-List.md       ← 确认当前任务进度
```

如果三个文件都不存在，提示用户先运行 easy-spec 生成规格。

---

## 记忆维护频率

| 文件 | 更新频率 | 触发方 |
|------|---------|--------|
| `context.md` | 每次会话结束 / 完成任务 / 发现阻塞 | 用户指令 |
| `Spec.md` | 规格变更时 | easy-spec 迭代模式 |
| `Task-List.md` | 任务状态变化时 | 用户指令 |
| Claude 全局记忆 | 发现持久价值信息时 | Claude 自动 |
