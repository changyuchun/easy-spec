# Spec 模板

对应 easy-product-builder 的 product-spec-template.md。
easy-spec 的核心产出文档，写清楚"要做什么、怎么运作、边界在哪里"。

---

## 1. 原始输入
- 需求名称
- 当前工作名（若正式名称未定）
- 一句话需求原文
- 任务类型：新建规格 / 迭代更新

## 2. 需求概述
- 需求名称
- 一句话定位
- 主要使用方（人 / 系统 / 服务）
- 核心目标
- 一期范围概述

## 3. 澄清纪要

### 已确认决策
- 已确认项

### 待确认项
- `[待确认]` 项

注意：只记录已追问触达但尚未定案的问题，不把空白直接塞进待确认。

### 被排除方案
- 明确不采用的方案及原因

### 关键假设
- 当前暂按什么假设推进

注意：只记录用户明确不确定、拒答，或授权 AI 先代补的内容。

### 术语定义
- 关键术语及定义

## 4. 使用方与场景

### 主要使用方
- 谁 / 哪个系统使用
- 使用前提

### 核心场景
- 场景 1
- 场景 2
- 场景 3

## 5. 范围

### v1 必做范围
- 一期必须成立的内容

### 非目标
- 当前明确不做的内容

### 后续版本
- 延后处理的内容

## 6. 核心对象与状态
- 主要实体
- 实体关系
- 关键状态
- 状态流转规则

## 7. 功能 / 行为设计

对每个主要功能写清楚：
- 名称
- 目的
- 触发方（角色 / 系统事件）
- 输入
- 输出
- 核心规则
- 边界情况

## 8. 业务规则
- 校验规则
- 状态流转规则
- 搜索 / 筛选 / 排序规则
- 删除 / 归档 / 恢复规则
- 权限规则
- 异常处理规则

## 9. 主要流程

按步骤写出一期主路径，例如：
- 触发 / 进入
- 核心操作
- 完成 / 结束

补充必要分支流程：
- 零状态流程
- 异常流程
- 回退流程

## 10. 与其他层的边界

### 与 UI 层（如有界面）
- 需要哪些核心页面
- 页面细节单独写入 `task-list.md`

### 与外部系统
- 上下游依赖
- 接口协议约定

## 11. 数据与系统约束
- 必需字段
- 派生字段
- 外部集成项
- 运行或部署前提

## 12. AI 功能（如适用）

只有需求真的包含 AI 能力时才写这一节。

每个 AI 功能至少写：
- 用户价值
- 触发位置
- 输入
- 输出
- 约束
- 失败处理

## 13. 开放问题
- `[待确认]` 项及其影响

## 14. 验收标准
- 用具体可判断的语句定义完成条件
- 标准对应行为，不对应实现方式
