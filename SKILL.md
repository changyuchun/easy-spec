# Easy Spec

## 显式加载规则

- 本 skill 只能由用户显式点名加载：`$easy-spec`、`easy-spec`，或清楚表达"加载/使用 Easy Spec skill"
- 不得因为用户提出需求描述、规格文档、技术方案或任务拆解相关提示而自动加载
- 如果用户没有显式点名，按普通 agent 流程处理，不应用本 skill 的追问和文档落盘流程

## 概述

这个 skill 的目标是把用户脑中"差不多有个方向"的一句话需求，逐步逼近成三类可执行文档，并维护整个开发周期的上下文：

- `.easy-spec/spec/Spec.md` — 功能规格（用户、场景、范围、流程、对象、规则、验收标准）
- `.easy-spec/spec/Task-List.md` — 开发任务清单（可执行、可追踪，1 人天粒度）
- `.easy-spec/memory/context.md` — 会话上下文（每次会话启动时读取，需求归档时清空）

需求完成时，将规格和复盘归档到 `.easy-spec/archive/` 下。

## 工作原则

- 核心方法是追问，不是脑补
- 进入规格设计前，先展示一份"探讨计划"，说明将从哪些方面推进；允许用户补充或修改，确认后再开始正式追问
- 最先确认的核心信息必须包含：需求名称；如果正式名称未定，也要先记录一个工作名，并标记是否待确认
- 默认采用"深挖但有限"的追问方式，先问干净必要的问题，再输出完整 spec
- 进入规格设计阶段后，每轮固定只问 3 个问题，不多不少；可以问很多轮，但每轮都要足够聚焦
- 追问与成文并行进行：每轮对话后都要立即更新草稿；下一轮提问前，先回看已完成文档与问题矩阵，再决定最该问的 3 个问题
- 未明确的信息可以先写入草稿，但必须标记为 `[待确认]` 或 `当前假设`
- `[待确认]` 只用于已经触达但尚未定案的问题
- `当前假设` 只用于用户明确不确定、拒答，或授权 AI 先代补的情况
- 停止追问的依据不是轮数，而是适用问题域都已经被触达并完成覆盖
- 只要是选择题，必须使用 `A/B/C/D` 形式标注选项；允许用户直接回复字母，不要求复制选项内容
- 用户答不清时，优先给 2-4 个 `A/B/C/D` 选项和简短取舍，推动其做决定

## 默认产物

所有文件统一输出到 `.easy-spec/` 目录：

```
.easy-spec/
├── spec/
│   ├── Spec.md               ← 核心规格文档
│   ├── Task-List.md          ← 开发任务清单
│   └── Spec-CHANGELOG.md     ← 仅用户明确要求时追加
├── memory/
│   ├── context.md            ← Level 0：当前会话进度（每次启动自动加载）
│   ├── index.md              ← Level 1：所有历史需求索引（每次启动自动加载）
│   └── summaries/            ← Level 2：每个需求的概要（按需加载）
│       └── <需求名称>.md
└── archive/                  ← Level 3：完整归档（显式加载）
    └── YYYY-MM-DD-<需求名>/
        ├── Spec.md
        ├── Task-List.md
        ├── Spec-CHANGELOG.md
        └── retro.md
```

明确不要默认生成：
- 独立的 PRD.md（功能规格已在 Spec.md 中）
- 独立的 Architecture.md（架构决策写入 Spec.md 的约束章节）
- 独立的 API-Spec.md（接口规范写入 Task-List.md 的接口任务说明中）

## 启动检查

开始时以 `.easy-spec/spec/` 作为默认规格目录。

目录规则：
- 所有规格文档默认存放在 `.easy-spec/spec/`
- 如果目录不存在，先创建该目录，再在其中生成或更新文档
- 非用户明确要求时，不把 spec 文档散落写到项目根目录

### 第一步：加载记忆（固定顺序）

```
1. 读取 .easy-spec/memory/context.md    → Level 0，了解当前进度
2. 读取 .easy-spec/memory/index.md      → Level 1，了解所有历史需求
   （index.md 不存在则跳过，说明是首个需求）
```

### 第二步：判断模式

| Spec.md | context.md | 判定 |
|---------|-----------|------|
| 存在 | 有内容 | **迭代模式**，context.md 已给出当前进度 |
| 存在 | 空或不存在 | **迭代模式**，规格存在但未进入开发 |
| 不存在 | 空或不存在 | **0-1 模式**，全新需求 |

### 第三步：历史关联感知（仅 0-1 模式）

加载了 index.md 后，根据用户描述的新需求，扫描 index.md 中所有历史条目：
- 对比新需求描述与历史需求的「一句话描述 + 核心技术/领域」
- 发现语义相关的历史需求（如上下游关系、同领域、技术复用）

发现相关时，**主动告知用户并询问**：
```
发现历史需求【用户标签系统】可能与本次相关（推送系统的上游依赖）。
要加载它的摘要作为背景参考吗？（推荐：是）
```

用户确认后，读取 `.easy-spec/memory/summaries/<需求名称>.md`（Level 2）。
Level 2 摘要作为背景参考，在追问和成文阶段自动融入，避免重复设计或踩历史的坑。

用户拒绝或无相关历史：直接进入追问阶段。

## 决策账本与渐进草稿

在多轮对话中，持续维护一份轻量决策账本，并把它同步沉淀到内部草稿中。

账本至少包含：
- 原始一句话需求
- 已确认决策
- 待确认项
- 被排除方案
- 关键假设
- 术语定义

每轮追问后都要在内部更新草稿：
- 把新确认的信息写入 `.easy-spec/spec/Spec.md` 对应章节
- 如果需要任务拆解，同步更新 `.easy-spec/spec/Task-List.md`
- 把尚未定案但已触达的问题标记为 `[待确认]`
- 把用户允许 AI 先代补的内容标记为 `当前假设`

每轮新提问前都要先做一次快速回看：
- 当前草稿已经确定了什么
- 当前草稿还缺什么
- 哪 3 个问题最能减少规格的不确定性
- 是否有问题已经可以不再重复追问

对用户侧的同步保持轻量，只包含：
- 当前理解
- 新锁定的决策
- 仍待确认的问题
- 下一轮要追问的主题

不要每轮都展示完整文档，完整稿只在问题矩阵覆盖完成后输出。

## 问题矩阵

进入成文前，先判断本轮需求会波及哪些问题域，并逐类覆盖。每个问题域都要在内部标记为：
- 已明确
- 已问未答
- 当前假设
- 不适用

只要某个适用问题域仍处于"未触达"，就不能输出完整文档。

### 规格问题矩阵

至少检查这些问题域：
- 需求名称 / 当前工作名
- 目标与成功标准
- 使用方（用户角色 / 上游系统）与使用场景
- v1 范围与非目标
- 主流程与关键分支
- 核心对象、字段、状态、状态流转
- 业务规则、优先级、触发条件
- 权限差异
- 异常、边界、失败处理
- 数据与系统约束、外部依赖

### 任务矩阵（只要有开发执行需求就进入）

- 是否需要数据库变更
- 是否有对外暴露的接口
- 是否有前端页面
- 关键非功能性要求（并发 / 延迟 / 可用性）
- 任务依赖关系
- 测试覆盖要求

## 工作流

### 0-1 模式

用于当前还没有明确规格文档的情况。

#### 第零步：展示探讨计划

在正式提问前，向用户展示一份简洁的探讨计划，至少包含：
- 本次准备从哪些方面推进
- 计划先问哪些核心信息
- 后续还会覆盖哪些主题
- 是否预计需要生成 Task List

计划展示规则：
- 计划要清楚，但不要写成长篇方法论
- 只有用户显式确认计划后，才进入正式规格设计阶段

#### 第一步：确认需求名称与核心骨架

最先确认：
- 需求或功能名称（正式名或工作名）
- 这是什么功能 / 改动
- 谁会使用（角色 / 上游系统）
- 用户要完成的核心任务是什么
- 一期必须覆盖的范围是什么
- 是否需要开发任务拆解

需求名称必须最优先明确，它会影响后续所有文档的命名和归档。

#### 第二步：建立问题矩阵

根据当前需求，判断哪些规格问题域和任务矩阵问题域适用。先列出内部缺口，不急着写完整正文。

#### 第三步：按主题追问并持续更新草稿

正式规格设计阶段的提问规则：
- 每轮固定只问 3 个问题
- 这 3 个问题应尽量属于同一主题或同一决策链路
- 如果要提供选项，必须标成 `A/B/C/D`

每轮结束后：
- 更新内部 `.easy-spec/spec/Spec.md` 草稿
- 对未定项标记 `[待确认]` 或 `当前假设`
- 向用户同步当前理解、已锁定项、剩余缺口和下一轮问题主题
- 再基于已更新的草稿，整理下一轮最值得继续问的 3 个问题

#### 第四步：输出完整 Spec.md

规格问题矩阵覆盖完成后，读取 [references/spec-template.md](./references/spec-template.md) 并生成完整 Spec。

成文规则：
- 写行为和规则，不写空话
- 优先沿用用户已经使用的业务术语
- `[待确认]` 只能标记已触达但未定案的问题
- `当前假设` 只能标记用户允许 AI 先代补的问题
- 不写实现代码、SQL DDL 或部署脚本

#### 第五步：按需生成 Task List

只有当需求包含明确的开发执行阶段时，才生成 Task List。

读取 [references/task-list-template.md](./references/task-list-template.md) 并生成任务清单。

任务拆解规则：
- 每个任务必须有明确的完成标准，具体可判断
- 每个任务控制在 1 人天以内
- 任务按执行依赖顺序排列，标明前置依赖
- 分组顺序：基础设施 → 数据层 → 业务逻辑 → 接口层 → 前端 → 测试

#### 第六步：初始化会话上下文

无论是否生成了 Task List，只要需求即将进入开发执行阶段，就读取 [references/session-context-template.md](./references/session-context-template.md) 并生成 `.easy-spec/memory/context.md`。

内容：需求名称、当前阶段、任务列表（如有 Task List 则全部标为"待开始"；如无 Task List 则写一条"开发中"占位）、无阻塞点、关键决策（来自本轮追问）、下一步建议。

**context.md 的后续更新**：由用户在每次会话中手动触发，例如：
```
T02 做完了，开始 T04，请更新 context.md
发现阻塞：T08 筛选方案未定，请记录到 context.md
```
easy-spec 不会自动感知进度变化，必须由用户明确触发更新。

### 迭代模式

用于已有 `Spec.md`、这次是做修改或扩展的情况。

#### 第一步：读取现有文档，识别变更

先读取 `Spec.md`，如果存在 `context.md` 也先读取，然后搞清楚：
- 这次到底改什么
- 是改核心流程、对象模型、业务规则，还是仅影响 Task List，还是两者都影响
- 新需求是否和现有文档冲突

#### 第二步：展示本轮修改计划

向用户展示一份简洁的修改计划：
- 判断哪些章节会受影响
- 需要追问哪些新问题
- 是否会同步影响 Task List

等用户确认后，再开始追问。

#### 第三步：识别受影响的问题域

判断这次变更会波及哪些规格问题域和任务矩阵问题域。

只要变更影响以下任一内容，就需要重新追问并覆盖对应问题域：
- 主流程、核心对象、状态或状态流转
- 业务规则、权限、异常处理
- 任务依赖关系（影响 Task List 的执行顺序）
- 验收标准

#### 第四步：边追问边更新草稿

围绕被波及的问题域继续追问，同步更新现有文档草稿。每轮固定只问 3 个问题。

只有在用户明确要求时，才追加 `Spec-CHANGELOG.md`。

#### 第五步：输出最终更新稿

被波及的问题域都已完成覆盖后，再输出最终版文档。

如果变更影响了任务依赖关系，同步更新 `Task-List.md`。  
如果变更影响了开发阶段的关键决策，提示用户更新 `context.md`。

### 归档模式

归档是一个**独立触发**的操作，不是 0-1 模式或迭代模式的线性步骤。

触发条件：用户明确说"这个需求做完了"、"可以归档了"、"帮我归档"。

读取 [references/archive-template.md](./references/archive-template.md) 后，按以下顺序执行：

**Step 1 — 创建 Level 3 归档**
在 `.easy-spec/archive/YYYY-MM-DD-<需求名>/` 下复制 Spec.md、Task-List.md、Spec-CHANGELOG.md（如有）。

**Step 2 — 生成 retro.md**
向用户追问复盘信息（允许说"先生成空模板"跳过）：
- 实际交付和规格的差异
- 哪些决策回头看是对的，哪些需要改
- 遗留了哪些技术债

**Step 3 — 更新 Level 1 索引**
在 `.easy-spec/memory/index.md` 末尾追加一行（文件不存在则先创建表头）。
字段：需求名称、完成日期、一句话描述、核心技术/领域、摘要链接。

**Step 4 — 生成 Level 2 摘要**
读取 [references/summary-template.md](./references/summary-template.md)，在 `.easy-spec/memory/summaries/<需求名称>.md` 生成摘要。
内容来源：Spec.md 的核心实体 + retro.md 的决策与教训 + context.md 的决策记录。

**Step 5 — 清空 context.md**
将 `.easy-spec/memory/context.md` 重置为空模板，等待下一个需求。

注意：`.easy-spec/spec/` 中的原文件**保留不动**，作为下次迭代的起点。

## 提问策略

优先问能逼出决策、能改变规格的问题，例如：
- 这个需求的名称最终叫什么
- 一期的主要使用方到底是谁
- 用户完成任务时第一步从哪里进入
- 创建时哪些字段必须填
- 什么条件会让对象进入激活 / 完成 / 失败状态
- 两条业务规则冲突时以谁为准
- 这个需求是否需要生成 Task List

避免低价值问题，例如：
- 让用户重复描述整个需求
- 在主流程未稳定前泛泛问"还有什么"
- 为了凑模板而追问不影响决策的细节
- 单轮抛出超过 3 个规格设计问题

## 记忆系统

读取 [references/memory-template.md](./references/memory-template.md) 了解完整的四级记忆体系设计。

四级概览：

| 级别 | 文件 | 内容 | 加载时机 | 写入时机 |
|------|------|------|---------|---------|
| Level 0 | `memory/context.md` | 当前会话进度 | 每次启动自动加载 | 每次会话更新，归档时清空 |
| Level 1 | `memory/index.md` | 历史需求索引（一行一个）| 每次启动自动加载 | 每次归档时追加一行 |
| Level 2 | `memory/summaries/<名>.md` | 需求概要（决策+实体+技术债）| 发现相关需求时按需加载 | 每次归档时生成 |
| Level 3 | `archive/<日期-名>/` | 完整归档（Spec+retro）| 用户显式要求时加载 | 归档时创建，只读 |

第五层：Claude Code 全局记忆（`~/.claude/projects/.../memory/`）负责跨需求、跨项目的用户偏好和经验，由 Claude 自动写入，不归 easy-spec 管理。

## 完成标准

当以下条件满足时，本 skill 完成一个完整循环：
- 需求可以用一句清楚的人话说出来
- 一期范围明确
- 主流程从头到尾写清楚了
- 核心对象、状态和规则已经定义
- 未解决问题被显式标注，而不是被偷偷跳过
- 如果需要开发执行，Task-List.md 已生成，每个任务都有可判断的完成标准
- `.easy-spec/memory/context.md` 已初始化，可供下次会话启动时加载
- 输出的文档可以直接交给下一阶段继续推进，不需要大量猜测
